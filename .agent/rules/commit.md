# コミット方針

**1つの論理的な変更単位ごとにコミットせよ。** 実装を全て終えてからまとめてコミットしてはいけない。

- 1つの関数追加、1つのコンポーネント、1つのバグ修正、1つの設定変更 — それぞれが1コミット
- ファイルを2-3個変更したら、コミットすべきタイミングかを自問せよ
- 未コミット変更が50行を超えたらフックが警告する。警告が出たら即座にコミットせよ

## コミットメッセージ形式

Conventional Commitに従う。日本語で簡潔に書く。

```
<type>: <タイトル — 何の変更を行ったかを自然文で簡潔に>

<理由 — なぜこの変更が必要だったか>
```

例:

```
feat: BigQueryデータセットをterraform化した

手動管理だったBQリソースをIaCに移行し、環境差分を防ぐ
```

typeは `feat` / `fix` / `refactor` / `chore` / `style` / `docs` / `test` を使い分ける。
