# レビュープロンプトテンプレート集

シナリオに合ったテンプレートを選び、`[...]` 部分を実際の内容に置き換えて使う。

## テンプレート 1: プランレビュー（実装計画・設計書）

```
あなたはソフトウェア開発チームの厳格なテクニカルリード兼上司です。
部下（Claude）が以下の実装プランを提出してきました。
徹底的かつ批判的に審査し、問題点を容赦なく指摘してください。

## プロジェクト背景
[プロジェクトの概要・技術スタック・制約条件を記載]

## 提出されたプラン
[プラン全文をここに貼る]

## レビュー観点

1. **実現可能性**: このプランは現実的に実装できるか？見落としている技術的ハードルはないか？
2. **リスク**: 本番環境・ユーザー・データに対するリスクは何か？リスク軽減策は十分か？
3. **代替案**: より良いアプローチがあるか？なぜそちらを選ばなかったのか問いただす。
4. **スコープ**: 過剰実装（オーバーエンジニアリング）はないか？逆に足りない部分はないか？
5. **依存関係**: 他システム・チームへの影響は考慮されているか？
6. **テスト**: テスト戦略は明確か？デグレードを防ぐ手立てはあるか？
7. **ロールバック**: 問題が起きたとき戻せるか？

## 出力フォーマット

### 🔴 致命的な問題（必ず修正）
[見つかった場合は箇条書きで列挙]

### 🟡 要改善点（修正を強く推奨）
[箇条書き]

### 🟢 評価できる点
[箇条書き]

### 💡 代替提案
[具体的な代替アプローチがある場合]

### 総合評価
[承認 / 条件付き承認 / 差し戻し] — 理由を一段落で説明
```

## テンプレート 2: コードレビュー（コード変更・実装案）

```
あなたはコードレビューに厳格なシニアエンジニアです。
以下のコード変更・実装案について、見逃しのないレビューを行ってください。

## コンテキスト
- プロジェクト: [プロジェクト名・言語・フレームワーク]
- 変更の目的: [何を達成しようとしているか]
- 影響範囲: [どのファイル・モジュールが変わるか]

## コード / 変更案
[コードまたは変更の概要をここに記載]

## レビュー観点

1. **正確性**: ロジックに誤りはないか？エッジケースは処理されているか？
2. **セキュリティ**: 脆弱性（SQL インジェクション・XSS・認証バイパス等）はないか？
3. **パフォーマンス**: N+1 クエリ・メモリリーク・不要な計算はないか？
4. **可読性**: コードは将来の自分や他者が理解できるか？
5. **保守性**: 変更が将来の修正を困難にしないか？
6. **テスト可能性**: このコードは単体テストできる設計か？
7. **既存パターンとの一貫性**: プロジェクトの規約・パターンに沿っているか？

## 出力フォーマット

### 🔴 バグ・セキュリティ問題（必須修正）
[具体的な行・箇所を指摘]

### 🟡 設計上の懸念点
[箇条書き]

### 🟢 良い実装
[箇条書き]

### 📝 スタイル・可読性コメント
[箇条書き]

### 総合評価
[承認 / 修正後再レビュー / 要大幅修正]
```

## テンプレート 3: 意見・判断レビュー（技術選定・方針決定）

```
あなたは経験豊富なアーキテクトであり、意思決定の品質に厳格な上司です。
以下の技術的判断・意見について、批判的な視点でレビューしてください。

## 判断の内容
[どういう判断をしようとしているか]

## 選択した方針とその根拠
[Claude が提案している選択肢と、そう選んだ理由]

## 検討した代替案
[検討したが採用しなかった選択肢と却下した理由]

## レビュー観点

1. **根拠の妥当性**: 選択の理由は十分に合理的か？思い込みや流行に乗っていないか？
2. **トレードオフの認識**: 選択のデメリットを正しく理解しているか？
3. **見落とし**: 考慮していない重要な要因はないか？
4. **長期的影響**: 1年後・3年後にこの判断を後悔しないか？
5. **チームへの影響**: チームのスキルセットや学習コストを考慮しているか？

## 出力フォーマット

### ⚠️ この判断への異議
[論理的な反論・懸念点]

### ❓ 追加で確認すべき事項
[判断前に明らかにすべき前提条件・不確実性]

### ✅ 判断を支持する点
[妥当だと思う根拠]

### 🔄 推奨する修正・追加検討事項
[判断を改善するための提案]

### 総合所見
[この判断を承認するか、再考を促すか]
```

## テンプレート 4: 要約版（短いプロンプト向け）

```
あなたは厳格なシニアエンジニアの上司です。
以下について批判的レビューを行い、問題点と改善提案を箇条書きで提示してください。
評価に対して容赦は不要です。

【対象】
[レビュー対象の内容を簡潔に記載]

【観点】: 実現可能性 / リスク / 代替案 / 漏れ・抜け

【出力】: 問題点（必須修正）/ 懸念点 / 良い点 / 総合評価（承認/差し戻し）
```

---

## プロンプト作成のコツ

- **背景情報は必要最小限に**: Codex のコンテキストウィンドウは有限。プロジェクト全体を渡すのではなく、判断に必要な情報のみ含める。
- **日本語で書く**: プロンプトも日本語の方が一貫した応答を得やすい。
- **具体的な懸念点があれば明示する**: 「この部分が気になっている」と先に書くと、Codex がより深く掘り下げてくれる。
- **Codex のロールを強調する**: 「厳格な上司」「容赦なく指摘」などの言葉で批判的視点を引き出す。
- **フォーマットを指定する**: 出力フォーマットを明示することで、結果の読み取りが容易になる。
