# Codex Review（厳しい上司レビュー）

Codex CLI (`codex exec`) を使って、Claude が作成したプラン・コード提案・意見を「厳しい上司」の視点で批評させる。このスキルの目的は、Claude の思考にバイアスや盲点がないかを外部視点で検証し、より堅牢な成果物を生み出すことにある。

## いつ使うか

次のいずれかに当てはまるとき、このスキルを使う：

- ユーザーが「Codexにレビューさせて」「上司に見てもらう」「ボスにチェックさせて」「/codex-review」等と言ったとき
- Plan モードで重要な実装プランを確定しようとしているとき（確定前にレビューを挟む）
- 設計上の意思決定を提案するとき（アーキテクチャ変更、技術選定、大きなリファクタリング等）
- ユーザーが Claude の意見に同意しているが、第三者視点での検証を求める空気があるとき

## ワークフロー

### ステップ 1: レビュー対象を整理する

レビューに渡す内容を一つのテキストブロックにまとめる。以下の情報を含める：

- **コンテキスト**: プロジェクト概要・背景（簡潔に）
- **レビュー対象**: プラン全文、または提案内容
- **判断の根拠**: なぜそのアプローチを選んだか
- **懸念点**: Claude 自身が気になっている点（あれば）

### ステップ 2: レビュープロンプトを構築する

`references/review-prompts.md` にシナリオ別のプロンプトテンプレートがある。レビュー対象の性質（プランレビュー・コードレビュー・意見レビュー）に合わせて適切なテンプレートを選んで使う。

プロンプトの構成原則：

- Codex に「厳しい上司」というロールを明確に与える
- レビューして欲しい観点を列挙する（実現可能性・リスク・代替案・コスト等）
- 出力フォーマットを指定する（問題点・改善提案・総合評価）
- 日本語で書く

### ステップ 3: Codex を実行する

スキル内のラッパースクリプトを使ってレビューを実行する。

```bash
echo "プロンプト内容" | .agent/skills/codex-review/scripts/run-review.sh -
```

内部では `codex exec -s read-only --ephemeral -o /tmp/codex-review-result.md` を実行する。結果は `/tmp/codex-review-result.md` に書き出される。

### ステップ 4: 結果を読み込む

Codex が完了したら `/tmp/codex-review-result.md` を読み込む。ファイルが空またはエラーが含まれている場合は、プロンプトが長すぎる可能性がある。その際はプロンプトを短縮して再試行する。

### ステップ 5: 結果を提示する

ユーザーに対して以下のフォーマットで結果を提示する：

```
---
## Codexレビュー結果

[Codex の出力をそのまま転記]

---
## Claude のコメント

[Codexのフィードバックに対する Claude の見解を記載]
- 同意する点: ...
- 再考が必要な点: ...
- 異議がある点（理由とともに）: ...
```

### ステップ 6: フィードバックを反映する

Codex の指摘を受けて、プランや提案を更新するか、更新しない場合はその理由をユーザーに説明する。指摘を無視したり、曖昧にスルーしてはいけない。すべての主要な指摘に対して明示的に応答する。

## エラー対処

- **Codex が応答しない**: ネットワーク接続やAPI認証を確認する。
- **出力ファイルが空**: プロンプトが長すぎる場合がある。レビュー対象を要約して再試行する。
- **`codex` が見つからない**: `which codex` で実際のパスを確認し、スクリプト内のパスを修正する。
- **タイムアウト**: `scripts/run-review.sh` のタイムアウト値（デフォルト180秒）を調整する。

## 注意事項

- Codex は外部プロセスであり、実行に数十秒かかる場合がある。ユーザーに待機を伝えること。
- レビュープロンプトに機密情報（APIキー・個人情報等）を含めないこと。
- Codex の指摘は参考情報であり、最終判断は Claude とユーザーが行う。
