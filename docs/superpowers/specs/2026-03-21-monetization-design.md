# 健常者エミュレータ事例集 収益化設計

## 背景と目的

健常者エミュレータ事例集は、社会生活の暗黙知を言語化・集積するUGCプラットフォームである。11,253件の投稿、517人の登録ユーザー、月間約195K PV（2026年3月時点、急成長中）の規模を持つ。

現在の収益はゼロ。過去にGoogle AdSenseを導入したが、収益が低くUXを損なう広告が多かったため撤退。投げ銭機能は存在するが機能していない。

**目標**: 月額30万円以上の持続可能な収益構造を構築する。

## 設計原則

1. **投稿の閲覧は無料を維持する** — サイトのミッション（知識格差の解消）と整合させる
2. **広告モデルは採用しない** — UXを損なう
3. **収益化は付加価値で行う** — 「読む」は無料、「より深く使う」「商用利用する」は有料
4. **自動化優先** — 運営の手間は最小限に。週数時間のコンテンツ制作は許容

## フェーズ1: 公開API + フリーミアムCLI + プレミアムサブスク

### 概要

プラットフォームへのプログラマティックなアクセス手段として公開APIとCLIを提供する。フリーミアムモデルで、基本機能は無料、追加機能をプレミアム（月額580円）で提供する。CLIの出力はJSON形式の構造化データとし、AIツール（Claude Code等）が加工する前提で設計する。

### 公開API（無料・認証不要）

事例検索APIをサイトのミッション（知識へのアクセス）の延長として公開する。

**エンドポイント**:
- `GET /api/posts` — 投稿一覧（ページネーション、ソート対応）
- `GET /api/posts/:id` — 投稿の詳細取得
- `GET /api/search?q=...` — 全文検索
- `GET /api/tags` — タグ一覧
- `GET /api/tags/:tagName/posts` — タグ別投稿一覧

**レスポンス形式**: JSON（AIが加工する前提で、human-readableである必要はない）

**Rate limit**: IP単位で60req/min（Cloudflare Rate Limiting Rules）

### CLI（フリーミアム）

npmパッケージとして配布。APIのラッパーとして動作する。

**無料機能**（APIキー不要、公開APIのみ使用）:
- 投稿の検索・取得
- タグ一覧・タグ別投稿取得
- 投稿の詳細表示

**プレミアム機能**（APIキー認証、月額580円）:
- **投稿者ダッシュボード**: 自分の投稿のパフォーマンス（PV、いいね、コメント、ブックマーク数）
- **高レートリミット**: 60req/min → 600req/min
- **プレミアムバッジ**: Web上の投稿・コメントにバッジ表示

**プレミアムAPIエンドポイント**（APIキー必須）:
- `GET /api/me/dashboard/summary` — 概要（累計投稿数、累計いいね数、累計閲覧数）
- `GET /api/me/dashboard/posts` — 投稿ごとのパフォーマンス一覧
- `GET /api/me/dashboard/highlights` — ハイライト（最も助けた投稿TOP3等）
- `GET /api/me/dashboard/trends` — 月別トレンドデータ

**ダッシュボードのデータソース**:
- PV: BigQuery（GA4）→ **日次バッチでD1に集計テーブルとして実体化**。API応答時はD1のみ参照
- いいね/コメント/ブックマーク: D1データベース
- **前提条件**: ユーザーと投稿の紐付け機能（後述）が必要

### 価格

- **月額580円**（税込）
- 単一プラン、年額プランなし
- プレミアムの課金開始は後から。まず無料のCLI+公開APIで利用者を獲得する

### 決済基盤

- **Stripe Subscriptions**: 月額定期課金
- **Stripe Customer Portal**: 解約セルフサーブ（初期は単一プランのため、主な用途は解約・支払い方法変更）
- **Webhook処理**:
  - Web Crypto API で署名検証（stripe-nodeのconstructEventはNode.js依存のためWorkers非互換）
  - イベントIDによる冪等性制御（D1に処理済みイベントIDを記録し重複処理を防止）
  - 処理対象イベント: `customer.subscription.created`, `customer.subscription.updated`, `customer.subscription.deleted`, `invoice.payment_succeeded`, `invoice.payment_failed`
  - D1書き込み失敗時: Stripeが自動リトリートするため、冪等性制御で安全に再処理

## フェーズ2: コンテンツライセンス販売

### 概要

投稿データを元にテーマ別キュレーション記事を運営側で制作し、全文無料公開する。記事を研修・教育目的で商用利用するためのライセンスをStripeで販売する。

**「読む」は無料、「使う」は有料**というモデル。

### テーマ別記事

#### コンテンツ構造

- 導入: テーマの概要、なぜこの暗黙知が重要か
- 事例: 投稿データから厳選した事例（要約+解説付き）
- 実践アドバイス: 事例から得られる具体的な行動指針
- 全文をSSRで公開（SEOクローラブル）

#### テーマ例

- 「職場で空気が読めないと言われたときの対処事例集」
- 「飲み会・会食のマナーで失敗しないための暗黙知」
- 「Discord・SNSでの暗黙のルール集」
- 「就活・面接で知っておくべき暗黙の了解」

#### 制作フロー

1. BigQuery + D1のデータから、いいね数の多い投稿をテーマ別に抽出
2. 運営が週数時間かけてキュレーション+解説を執筆
3. サイト上に公開
4. 月1〜2本のペースで積み上げ

### ライセンス販売

#### 想定顧客

- 企業の人事・研修部門（コミュニケーション研修、新人研修）
- 就労支援事業所・福祉施設
- 学校・教育機関
- キャリアコンサルタント、カウンセラー

#### ライセンスで許諾される権利

- 社内研修・教育目的での複製・配布
- スライド・教材への引用・転載
- PDF化して社内共有
- **許諾されないもの**: 再販、外部公開、改変後の再配布

#### 価格体系

| プラン | 内容 | 価格 |
|---|---|---|
| 単品 | 1記事のライセンス | 5,000円 |
| テーマパック | 1テーマの全記事 | 30,000円 |
| 年間フリー | 全記事使い放題（1年） | 月額15,000円 |

#### ユーザー導線

```
Google検索「職場 暗黙のルール」
  → テーマ別記事ページ（全文無料で読める）
  → 記事下部にCTA:「この記事を研修・教育で使いませんか？」
  → ライセンス購入ページ（Stripe Checkout）
  → 決済完了 → ライセンス証書PDF + 利用規約を自動メール送付
```

#### 技術的な実装

- **決済**: Stripe Checkout Session（単品決済 + 月額サブスクリプション）
- **PDF生成**: `pdf-lib` を Workers 内で実行（軽量なライセンス証書程度なら128MB制限内で可能）。証書にはライセンスID、組織名、対象記事、有効期限を記載
- **メール送信**: Resend（Cloudflare Workersとの相性が良く、REST APIで呼び出し可能）
- **ライセンス管理**: D1にライセンステーブル（ライセンスID、組織名、メール、対象記事/プラン、有効期限、Stripe決済ID）
- 記事ページ自体にはアクセス制限なし

### SEO効果

- 全文無料公開のため、一般キーワード（「職場 暗黙のルール」等）での検索流入を獲得
- 記事からサイト本体への内部リンクで投稿への導線も構築
- 積み上げ型でロングテール流入が増加
- SEO効果が出るまで3〜6ヶ月はかかる前提で計画する

### 収益目標

フェーズ2はSEO効果と記事の積み上げに依存するため、立ち上がりは遅い:

| 時期 | 記事数 | 月間売上（想定） |

| 6ヶ月後 | 10〜15本 | 5〜15万円 |
| 12ヶ月後 | 20〜30本 | 15〜30万円 |

## 技術的な前提条件・新規実装が必要なもの

### 1. Stripe連携基盤

- Checkout Session（サブスク + 単品決済）
- Subscription管理
- Webhook（Web Crypto APIで署名検証、冪等性制御）
- Customer Portal

### 2. ユーザーと投稿の紐付け

現在、投稿は `postAuthorIpHash` で匿名化されておりユーザーIDと紐付いていない。

- `dim_posts` に nullable な `userId` カラムを追加（型は `integer` — `dim_users.userId` への外部キー参照）
- 注: コードベース内で `now_editing_pages.userId` や `fct_post_edit_history.editor_user_id` は `text` 型だが、`dim_users.userId` は `integer` 型。新規カラムは `dim_users.userId`（integer）を参照する
- 新規投稿: ログイン状態であれば `userId` を自動記録。匿名投稿も引き続き可能
- 既存投稿: 紐付けなし（遡及しない）。将来的に「自分の投稿を紐付ける」機能を検討可能だが、初期スコープ外
- 投稿の表示上は引き続き匿名（IPハッシュ表示）。`userId` はダッシュボード用の内部データとしてのみ使用

### 3. プレミアム状態管理

- D1にサブスクリプション状態テーブル（userId（integer）、stripeCustomerId、stripeSubscriptionId、status、currentPeriodEnd）
- ローダー/ミドルウェアでプレミアム判定
- **支払い失敗時の挙動**: `invoice.payment_failed` 受信後、即座にプレミアムを停止せずStripeのDunning（自動リトライ、約3週間）に委ねる。`customer.subscription.updated` で `status` が `past_due` → `canceled` に変わった時点でプレミアムを停止する。`currentPeriodEnd` までは猶予期間としてプレミアム機能を維持

### 4. 公開API

- 既存の検索機能をREST APIとして公開（認証不要）
- Rate limit: IP単位で100req/min（Cloudflare Rate Limiting Rulesで実装）
- **Stripeシークレットの保管**: Cloudflare Secrets Store を使用（コードベースの既存パターンに合わせる）

### 5. プレミアムAPIキー発行

- プレミアムユーザーにAPIキーを発行（ダッシュボードCLI認証用）
- D1にAPIキーテーブル（userId、apiKey、createdAt）
- APIキーはStripeサブスク状態と連動（解約時に無効化）

### 6. 投稿者ダッシュボード

- PV集計の日次バッチ: Cron Trigger → BigQuery集計 → D1書き込み
- D1に `post_daily_stats` テーブル（postId、date、pageviews）
- **バッチ設計**: 前日分のPVのみを差分取得（全件フルスキャンしない）。GA4の `events_*` テーブルを前日の `_TABLE_SUFFIX` で絞り、`page_location` から `postId` を抽出してGROUP BY。結果は数百〜数千行程度でWorkers Cron Triggerの30秒CPU制限内に収まる
- フロントエンド: モバイルファーストのグラフ表示

### 6. ライセンス管理

- D1にライセンステーブル
- pdf-lib でライセンス証書PDF生成（Workers内）
- Resend でメール送信

### 7. テーマ別記事ページ

- 新規ルート（`/guides/:slug` 等）
- 記事コンテンツはD1に保存（タイトル、本文Markdown、テーマ、公開日、関連投稿ID）
- 記事下部にライセンス購入CTA

## リスクと対策

### UGCの商用ライセンスに関する法的リスク

投稿者が匿名で共有した経験が商用ライセンスとして販売されることへの懸念がある。

**対策**:
- キュレーション記事は投稿をそのまま転載するのではなく、複数の投稿から得られた知見を**運営が再構成・解説したオリジナルコンテンツ**として制作する
- 個別の投稿を特定できる形での引用は避け、パターンや教訓としてまとめる
- 利用規約にコンテンツの二次利用に関する条項を明記し、既存ユーザーに告知する
- 投稿者からオプトアウト（自分の投稿をキュレーション対象外にする）の仕組みを用意する

### 法的コンプライアンス

- **特定商取引法に基づく表記**: サブスクリプション・デジタルコンテンツ販売に必要。事業者名、住所、連絡先、価格、解約方法等を記載したページを新設
- **インボイス制度**: 適格請求書発行事業者の登録状況を確認。法人向けライセンス販売では適格請求書の発行が求められる可能性がある
- **資金決済法**: 投げ銭機能が既に存在するが、ライセンス販売は役務提供の対価であり資金決済法の対象外

### チャーン（解約）リスク

プレミアム機能が3つと薄いため、新規性が薄れた後の解約率が高くなる可能性がある。

**対策**:
- ローンチ後のユーザー行動データを見て、追加機能を検討する（例: 投稿の下書き保存、予約投稿、コメント通知カスタマイズ等）
- 投稿者ダッシュボードは継続的に更新されるデータなので、定着効果が期待できる
